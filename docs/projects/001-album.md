# プロジェクト 001: アルバム機能（しおり）

## 1. 概要

「ひとさじ」アプリに、食事の場面に添えた一言メッセージ（将来的には写真も）を時系列に記録・閲覧できるアルバム機能を追加する。

最終ゴールは「撮影 → 一言 → 記録 → 後日見返す / リマインド」の体験。本ドキュメントでは段階的な PR 分割と、最初に取りかかるアルバム画面のレイアウト方針を整理する。

## 2. アプリコンセプトとの整合

`appname-concept/README.md` より抜粋した、本機能に直接関わる指針:

- **介在価値**: あたたかい記憶の蓄積（プレッシャーのない「**しおり**」機能）
- **タスク感の排除**: 連続記録（ストリーク）を作らず「忘れてもいい、いつ戻ってきてもいい」寛容さ
- **手触り感**: 紙やリネンのような「アナログな暖かさ」
- **蓄積の美**: 記録が増えるほど画面が豊かになる「静かなご褒美」

→ アルバムは Instagram や Google フォトの「メディア管理ツール」ではなく、**「ひとさじ」のコンセプト『しおり』を体現する場**として設計する。

## 3. 機能要件（最終形）

### コア体験

1. ホーム画面から「しおりを残す」アクションを起動
2. 撮影 → 一言メッセージ入力 → 保存
3. アルバム画面で、しおりを時系列に閲覧
4. 定期的に過去のしおりをリマインド（On this day 体験）

### 将来候補（コアの後で検討）

- **アルバムレイアウトの選択（設定機能）**: 採用 Option A を既定としつつ、設定画面でユーザーが Option E（手帳/ノート型）や Option I（スクラップブック型）に切り替えられるようにする。
    - 動機: コンセプト全体は「謙虚で控えめ」を志向するが、ユーザーごとに「もっと装飾的にしたい / もっと素朴にしたい」という好みがあり得る。「強要せず、選べる」というコンセプトの寛容さとも整合する。
    - 装飾レイヤー（§6.4 の F: 季節の背景、G: 蓄積の器、マスキングテープ）も、設定で個別に ON/OFF できるとよい。
    - これにより、本ドキュメントで一旦不採用とした E や I の世界観を、捨てずに取っておける。

## 5. PR-1 のスコープ（完了）

**目的: アルバム画面の見た目をダミーデータで体感できるようにする**

実際に含めたもの:

含まなかったもの（後続 PR で対応）:
- 入力フォーム画面
- 永続化
- ホーム画面（残り食事数表示）との接続 — **他メンバーが開発中のためコンフリクト回避**

## 6. レイアウト案の比較

検討中の案を「①レイアウト本体」「②本体に重ねる装飾レイヤー」「③検討から外した案」に分類して整理する。

### 6.1 レイアウト本体の案

#### Option A: しおり風タイムライン

月見出しの下に、紙のような柔らかいカードが縦に並ぶ。カードの中身は「日付（小さく薄く）＋一言メッセージ」。

```
┌─────────────────────────┐
│  2026 年 5 月            │
│                         │
│  ┌─────────────────────┐│
│  │ 5/3                  ││
│  │ 駅前の立ち食いそば。  ││
│  │ ネギ多め。           ││
│  └─────────────────────┘│
│  ┌─────────────────────┐│
│  │ 5/2                  ││
│  │ スーパーの半額シール ││
│  │ 弁当、助かった。      ││
│  └─────────────────────┘│
│                         │
│  2026 年 4 月            │
│   ...                    │
└─────────────────────────┘
```

- ✅ 「しおり」のメタファーがそのまま視覚化される
- ✅ 写真がないフェーズ 1 でも余白と紙感で成立する
- ✅ 写真追加時はカードに画像エリアを足すだけで自然に拡張可能
- ✅ タスク感がなく、月単位で「ふと開く」体験に合う
- ✅ 既存 Welcome 画面・ホーム画面モックのデザイン言語の自然な延長

#### Option B: Instagram 縦長フィード型

写真を主役にした大きなカードが縦に並ぶ。

```
┌─────────────────────────┐
│ ┌─────────────────────┐ │
│ │  [写真エリア]         │ │
│ │                     │ │
│ ├─────────────────────┤ │
│ │ 2026/05/03           │ │
│ │ 駅前の立ち食いそば。  │ │
│ └─────────────────────┘ │
│ ...                     │
└─────────────────────────┘
```

- 写真前提なら強いが、テキストのみだと余白が間延びしがち
- 「メディアを見せる」志向で、しおりの静けさには合わない

#### Option C: Google フォト グリッド型

写真サムネイルが密に並ぶ。

- 写真がないと完全に成立しない
- コンセプトの「アナログな暖かさ」とは方向性が逆（情報密度を上げる UI）

#### Option D: ノート/ジャーナル型（テキスト中心リスト）

日付見出しの下に短文が連なる、ToDo に近い表現。

- 情報密度は最も高い
- ただし「タスク感」が出やすく、コンセプトと相性が悪い

#### Option E: 手帳/ノート型

画面全体の背景を「開いた手帳/ノート」として描き、エントリは罫線の上に書かれた日記のように並ぶ。Option A の構造（時系列カード）を「ページに書き込まれた」表現に置き換えた発展案。

```
┌──────────────────────────┐
│ ▒▒▒ 紙のテクスチャ ▒▒▒    │ ← 背景はクリーム/紙系
│                          │
│  ━ 2026 年 5 月 ━━━       │ ← 月見出し（インデックスタブ風）
│  ───────────────          │ ← 罫線
│  5/3 駅前の立ち食いそば。  │
│      ネギ多め。            │
│  ───────────────          │
│  5/2 スーパーの半額シール │
│      弁当、助かった。      │
│  ───────────────          │
│                          │
│  ━ 2026 年 4 月 ━━━       │
│  ───────────────          │
│  ...                     │
└──────────────────────────┘
```

- ✅ コンセプトの「アナログな暖かさ」「タスク感の排除」と強く整合
- ✅ 縦スクロールの「ページが続くノート感覚」を、横スワイプの操作性問題なしで実現
- ✅ 写真追加時、ページに貼られたポラロイドのような表現に拡張可能（マスキングテープ風レイヤーと特に相性良）
- ⚠ ノート感が強すぎると装飾が主役を取るリスク
- ⚠ 既存デザイン言語（白基調・薄影・水色アクセント）と方向性が異なるため、画面間の不連続が出る恐れ

#### Option I: スクラップブック / マスキングテープ型

各エントリを「紙片」として扱い、マスキングテープで台紙に貼られているような表現。Option E が「整然と日記が並ぶ静謐感」だとすれば、こちらは「ばらばらと貼られたコラージュ的な愛着感」。

```
┌──────────────────────────┐
│ ▒▒▒ コルクボード/壁紙 ▒▒▒  │ ← 台紙
│                          │
│      ▓▓        ▓▓        │ ← マスキングテープ
│   ┌──────────────┐       │
│   │ 5/3            │     │
│   │ 駅前の立ち食い   │     │
│   │ そば。ネギ多め。 │     │
│   └──────────────┘       │
│                          │
│         ▓▓        ▓▓     │
│         ┌──────────────┐ │
│         │ 5/2          │ │
│         │ スーパーの   │  │
│         │ 半額シール...│  │
│         └──────────────┘ │
│                          │
│   ▓▓        ▓▓           │
│   ┌──────────────┐       │
│   │ ...          │       │
└──────────────────────────┘
```

- ✅ アナログ感は全案中で最も強い（テープ・紙片・台紙の三層構造）
- ✅ 写真追加時、紙片がそのままポラロイドに置き換わる自然な拡張
- ✅ 一枚一枚が「思い出のスクラップ」として独立した存在感を持つ
- ⚠ 実装コスト高（傾きランダム、配置ロジック、紙のテクスチャ等）
- ⚠ ランダム配置のため「1 ヶ月分の俯瞰」「日付検索性」が下がる
- ⚠ 装飾が前面に出すぎる傾向。控えめな演出にしないとコンセプトの寛容さを損なう恐れ

E と I のトーン対比:

- **Option E**: 静謐、日記的、整然 — 過去を「ふと開いて読む」体験向き
- **Option I**: にぎやか、コラージュ的、愛着 — 過去を「並べて眺める」体験向き

#### Option H: 箸袋 / 献立カード型

1 件 = 1 枚の食事関連アイテム（箸袋・お品書き・献立カードなど）として表示。「ひとさじ＝食」の世界観に最も濃く寄せる発想。

- ✅ 食のテーマを最も強く打ち出せる
- ✗ テキストのみのフェーズでは空気を作りにくい
- ✗ 装飾が前面に出すぎ「世界観の強制」になりやすい（タスク感とは別の意味で寛容さを損なう恐れ）

### 6.3 Option A を採用した理由（観点別評価）

複数案を比較検討し、一度は Option E を推奨に置いたが、客観的に再評価した結果 Option A に着地した。決め手となった観点を以下にまとめる。

#### ① 既存のデザイン言語との一貫性

`app/index.tsx`（Welcome 画面）と現状検討中のホーム画面モックは、どちらも「白背景・薄い影・丸みのあるカード・薄い水色アクセント」というミニマル＋柔らかい一貫したデザイン言語を採用している。ホーム画面モックの吹き出しカードは、白いカードが浮かんでいる時点ですでに "しおり" の現代的な解釈になっている。**この語彙の自然な延長としてアルバムを乗せる**のが最も無理がなく、E や I は新しい視覚パラダイムを画面に持ち込んでしまうリスクがあった。

#### ② コンセプトの「字義」と「精神」を分けて読む

`appname-concept/README.md` には「アナログな暖かさ」「しおり」といった字義的な手がかりがある一方、その精神を支える指針として:

- 「ひらがなの柔らかな響きによる『道具』としての**謙虚さ**」
- 「ユーザーに大それた変化を強いない、**心理的ハードルの低減**」
- 「**プレッシャーのない**『しおり』機能」
- 「**タスク感の排除**」

がある。これらはどれも装飾を**減らす**方向を示している。E のノート罫線・インデックスタブや、I のテープ・台紙・紙片・ランダム配置は、字義には合うが精神には逆向きになる可能性がある。

#### ③ スキューモーフィズムのリスク

E（ノート）と I（スクラップブック）はどちらも物理オブジェクトを画面上で再現する典型的なスキューモーフィズム表現。これらは:

- 「やりすぎ」で kitsch に転びやすい
- iOS 7 以降のフラットデザイン以来、現代的な OS の流儀から離れる
- Expo / React Native の標準スタックでテクスチャ表現を凝ると、プラットフォーム間で見た目がブレやすい

#### ④ 「アナログな暖かさ」は装飾ではなく余白と影で表現できる

Welcome 画面はすでにそれを実演している:

- 柔らかい影 (`shadowColor: #7E8AAB, opacity: 0.15, radius: 12`)
- 大きな余白
- 押し付けがましくない水色アクセント

Option A はこの語彙の自然な延長で、「アナログな暖かさ」を **装飾の追加ではなく抑制の徹底** で表現する。

#### ⑤ 「蓄積の美」は別の手段で上品に表現できる

コンセプト原文の「蓄積するほど画面が豊かに」は、§6.4 の Option G（蓄積の器/盆栽）のような小さな常駐演出で表現する方が、レイアウト全体を装飾するより上品。器が少しずつ豊かになる、という静かなご褒美。これは A にも E にも I にも乗せられる装飾だが、**ベースが静かな A の方が変化に気づきやすい**。

#### ⑥ Phase 1（テキストのみ）の安定感

A は写真がなくてもテキスト一行で成立する。E は罫線で空気を作れる。I は紙片だけが並ぶとやや寂しい。**A はテキストでも写真でも美しく見せられる、最も寛容な土台**。

#### トレードオフの整理

| 観点 | A vs 他案 |
|---|---|
| 既存デザイン言語との整合 | A ≫ E > I |
| コンセプトの「精神」（謙虚さ・抑制） | A > E > I |
| コンセプトの「字義」（ノート・スクラップ） | I > E > A |
| Phase 1 の安定感 | A ≧ E > I |
| 実装コスト | A < E < I |
| 写真追加時の自然さ | A ≒ E ≒ I |
| 「蓄積の美」装飾との相性 | A > E > I |

「字義」だけを見れば I > E > A だが、「精神 × 既存デザイン × コスト × 安定感」の総合では A が最もバランスが良い。

#### A を選んだ後の戦略

E や I の良さを A の上に **選択的に積み上げる** ことで、字義のニュアンスも取り込める:

- A + クリーム背景（§8 の検討余地）→ 軽い「ノート感」が出せる
- A + マスキングテープ装飾（§6.4）→ 写真追加時にスクラップ感が出る
- A + Option G（蓄積の器、§6.4）→ コンセプトの「蓄積の美」を直接表現
- 将来のレイアウト切替設定（§3）→ E や I を「選びたいユーザーには選ばせる」道も残す

### 6.4 装飾レイヤー（本体に重ねる）

レイアウト本体とは独立して、世界観を補強する装飾を後から段階的に足せる。Phase 1 では基本的に入れない（最小スコープ優先）。

#### Option F: 季節の背景

月ごとに背景の色味が四季を反映する。コンセプト原文の「光と影、季節の移ろいを背景の変化で表現」に直結。

- 春: 淡い桜色 / 夏: 青緑 / 秋: 山吹 / 冬: 薄いブルーグレー
- 採用 A の白背景に薄く重ねると、月をまたぐ瞬間に空気が変わる体験になる
- 強すぎると「謙虚さ」を損なうので、ごく薄く重ねる前提

#### Option G: 「蓄積の美」を可視化する小さな器/盆栽

画面の片隅（例: 上部）に小さな器や植物のイラストがあり、しおりが増えるほど豊かになる。コンセプト原文の「蓄積するほど画面が豊かに」に直結。

- ホーム画面の「残り食事数」(=減るもの) に対する「これまで残したしおり」(=蓄積するもの) の対称表現になり、アプリ全体の核演出になりうる
- アルバム画面単体ではなく、共通ヘッダー的に組み込む案
- 採用 A のミニマルなベースの上で、少しずつ豊かになる器の変化が際立つ

#### マスキングテープ風（写真の角飾り、軽量版）

Option I（スクラップブック型）の "弱い" 適用版。写真追加時（PR-6）、写真の左右上にマスキングテープ風の小さな矩形を重ねる。**採用 A のカードに重ねて、ポラロイドが手帳に貼られているような温度感** を作る。

将来 Option I をレイアウト切替設定（§3）で選べるようにする場合、I 側は本体に既にテープ表現が組み込まれているためこの装飾は不要。

### 6.5 検討から外した別案

#### ページめくり型（横スワイプ）

本のページのように左右にスワイプして月をめくる。体験は印象的だが「逆向きに捲るのが大変」「スクロール性能とアクセシビリティに課題」のため不採用。**Option E に「縦スクロールのページ感覚」として吸収**した。

### 6.6 写真追加時のレイアウト方針（決定）

カード/ページ内の要素の並びは **「日付 → 一言メッセージ → 写真」** の順とする（= 写真より先に一言が来る）。理由:

- **「一言＝声」「写真＝物証」のヒエラルキー**: ひとさじでは一言メッセージが「過去の自分の声」、写真は記憶を裏付ける物証。声が先に立ち、写真がそれを受ける構図が、コンセプトの「立ち止まるための鏡」と整合する。
- **Instagram の文法を意図的に反転**: 写真主役で文を従える Instagram の語法と差別化することで、「メディアを楽しむ場」ではなく「自分の声を残す場」だというメッセージになる。
- **読み返し体験での余韻**: 後で開いた時、まず短い一言が目に入って想起のフックになり、写真がそれを補強する流れの方が「映像→文字」より静かな余韻を作る。
- **手紙/日記の伝統的な構造**: 日付・本文・添付（写真）という順は、はがきや日記の標準フォーマットと一致する。

実装ガイド: カード/ページ内のレイアウトは `[日付] / [一言メッセージ（主役、フォントやや大きめ）] / [写真（あれば）]` の縦並び。

### 7.3 採用方針と PR-1 の結果

**当初方針（PR-1 開始時）: ① useState を継続**

- スコープを「アルバム画面 + ダミーデータ + 双方向の導線」に絞り、レビュー負担を最小化
- 3 画面までは `App.tsx` の if/else で十分管理可能
- ナビゲーション基盤の入れ替えと画面追加を同 PR に乗せると、レビューが「機能追加 + インフラ変更」の二本立てになって辛い

**サイズベース判断方針**: PR の修正行数を見て、小さく収まれば同一 PR 内で ③ expo-router への移行を同時に行う余地を残していた（4 画面目追加時の手戻り回避を狙ったもの）。

**実際の結果: ③ expo-router 構成に合流（main 側で先行）**

PR-1 実装後の `git pull origin main` 時に、main 側で先行していた他メンバーの expo-router 移行（`screens/` → `app/`、`App.tsx` 削除、各画面の router 化）と衝突した。当初の「サイズベース判断」は適用されず、main 側の先行移行に追従する形でアルバム画面を `app/album.tsx` として配置し、`router.back()` ベースの戻り遷移に切り替えてマージ解消した。

→ **PR-1 は最終的に ③ expo-router 構成で完了**。本来想定していた「PR-1 後に別 PR で ③ へ移行」のシナリオは発生せず、4 画面目（PR-2 のフォーム）以降は `app/*.tsx` を増やしていく形で進められる。

**今後の方針: ③ expo-router 上で画面追加**

- PR-2 以降は `app/*.tsx` を追加し、`app/_layout.tsx` の Stack に登録するとナビゲーションに組み込まれる
- typed routes（自動生成される型）を活用していく
- 過去の選定理由（Expo 公式デフォルト / ファイルベース / Web URL 標準 / TypeScript 型自動生成）は引き続き有効
